## Objective

Promote board context from display-only state into issue browsing scope for Scrum boards. Selecting a Scrum board should narrow `/jira-issues` to the board's active sprint when one exists.

## Context

Jira's practical workflow is often:

```text
project > board > sprint/filter > issues
```

M3.1/M3.3 established active project, focused issue, faceted filters, and a consolidated cockpit. M3.5 builds on the M3.4 board picker/active board foundation and makes Scrum board selection useful for narrowing issue search.

## Scope

M3.5 includes:

1. Fetch active sprint for selected Scrum board.
2. Use active sprint as `/jira-issues` default scope when available:

```jql
sprint = <activeSprintId> AND statusCategory != Done ORDER BY updated DESC
```

3. Compose faceted filters with active sprint scope.
4. Fall back gracefully to project scope when no active sprint exists or sprint lookup fails.

M3.5 depends on M3.4 for `/jira-boards`, active board runtime context, session restore, cockpit display, and `jira_get_current_context` board reporting.

## Non-goals

- Kanban saved filter scoping. Planned for M3.6.
- Board column/swimlane rendering.
- Moving issues across board columns.
- Creating/closing sprints.
- Persisting active board to local config.
- Jira writes or LLM-facing write tools.

## API candidates

```text
GET /rest/agile/1.0/board/{boardId}/sprint?state=active
```

Use Jira Server/Data Center-compatible shapes where possible.

## UX

After selecting a Scrum board with M3.4 `/jira-boards`, `/jira-issues` applies active sprint scope when available:

```text
Jira · CHATAPP · Board iOS Scrum Board
Filter: Sprint 78 · not done
Issue: none
Actions: /jira · /jira-issues · /jira-refresh
```

### `/jira-issues`

If active Scrum board has active sprint, default scope becomes:

```text
Sprint 78 · not done
```

Faceted filters then refine within active sprint scope.

## Fallback behavior

If sprint lookup fails or no active sprint exists:

- Keep board context selected.
- Show status/warning in cockpit.
- `/jira-issues` falls back to active/configured project scope.
