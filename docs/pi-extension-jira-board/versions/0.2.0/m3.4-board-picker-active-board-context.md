## Objective

Add foundational board support before implementing Scrum or Kanban-specific board scoping. Users should be able to browse boards for the active project, select one, and see that board reflected in the Jira cockpit and current context.

## Context

The intended Jira context hierarchy is:

```text
project > board > sprint/filter > issues
```

Before implementing Scrum active sprint scope or Kanban saved filter scope, the extension needs a shared board foundation:

- Board listing.
- Board picker UI.
- Active board runtime/session state.
- Cockpit/current-context rendering.

This prevents Scrum and Kanban milestones from each reimplementing board selection/state.

## Scope

M3.4 includes:

1. Add `/jira-boards [query]` interactive board picker.
2. List boards for active/current project.
3. Filter/page board results.
4. Select/apply board as active board context.
5. Persist active board context in session custom entries.
6. Restore active board context on session start.
7. Show active board in cockpit widget.
8. Return active board in `jira_get_current_context`.
9. Keep `/jira-issues` project/facet scoped for now unless an explicit JQL is provided.
10. Optionally bootstrap active board id from configured `JIRA_BOARD_ID` when available, but do not require board name lookup for M3.4 completion.

## Non-goals

- Scrum active sprint issue scope. Planned for M3.5.
- Kanban saved filter issue scope. Planned for M3.6.
- Board columns/swimlanes.
- Moving issues across board columns.
- Sprint creation/closing.
- Persisting active board to local config.
- Jira writes or LLM-facing write tools.

## API candidate

```text
GET /rest/agile/1.0/board?projectKeyOrId=<projectKey>&startAt=<n>&maxResults=<n>
```

If Jira Server/Data Center board search support differs, implement a safe compatible fallback and document the observed behavior.

## UX

### `/jira-boards`

Example browser:

```text
Jira boards · CHATAPP

> iOS Scrum Board       scrum
  Web Kanban Board      kanban
  Release Board         kanban

↑↓ select · enter apply · n/p page · / filter · esc cancel
```

On select:

```text
Jira · CHATAPP · Board iOS Scrum Board
Filter: not done
Issue: none
Actions: /jira · /jira-issues · /jira-refresh
```

`/jira_get_current_context` should report:

```text
Project: CHATAPP 聊天APP
Board: iOS Scrum Board (#123, scrum)
Filter: ...
Focused issue: ...
```

## State model

Minimal board state:

```ts
interface ActiveJiraBoard {
  id: number;
  name?: string;
  type?: "scrum" | "kanban" | "simple" | string;
  projectKey?: string;
}
```

This should be stored alongside existing active project/filter/focused issue context.
