## Objective

Make M3.1 focus state useful to the agent without injecting Jira state into every LLM turn. Users should be able to focus a Jira issue/project/board in the browser and later refer to the current Jira context naturally, e.g. “inspect focused issue” or “current project backlog”.

## Design decisions

- Do **not** use a `context` hook for Jira focus state.
- Do **not** inject current Jira state into every prompt.
- Persist focused/current context as session custom entries so it can survive reload/compaction and be restored from the active branch.
- Expose focused/current context through explicit read-only tools only.
- Keep board support minimal in M3.1; board picker is optional/deferred to M3.3.

## Current context hierarchy

Jira context should be modeled as:

```text
project > board > issue filters > focused issue
```

Where:

- `project` is the broadest active scope.
- `board` is optional but can narrow scope through active sprint or board filters.
- `issue filters` further refine issue browser scope.
- `focused issue` is the current direct work target.

## Scope

M3.1 addendum includes:

1. Extend runtime context with:
   - active project
   - active/configured board id
   - active issue filter summary/state when available
   - focused issue compact metadata
2. Persist context changes with session custom entries.
3. Restore context from the current session branch on `session_start`.
4. Add read-only tool `jira_get_current_context`.
5. Add read-only tool `jira_get_focused_issue`.
6. Make `/jira-issues` focus action update focused issue state.
7. Include configured board id in current context when no active board picker exists.
8. Keep all Jira writes unchanged and safe.

## Non-goals

- No context hook injection.
- No clickable widget behavior.
- No board picker in M3.1.
- No saved default project/board config mutation.
- No full Jira metadata system.
- No new Jira write behavior.

## Tools

### `jira_get_current_context`

No parameters.

Returns compact state only:

```text
Current Jira context:
Project: CHATAPP 聊天APP
Board: 123
Filter: CHATAPP · not done · fixVersion:v1.62 · component:IOS
Focused issue: CHATAPP-5410 [iOS] ...
```

Expected use:

- User says “current project”, “this board”, “focused issue”, or “current filter” without naming keys.
- Agent needs to understand current Jira scope before choosing another Jira tool.

### `jira_get_focused_issue`

No parameters.

Behavior:

- If no focused issue exists, returns a clear error.
- If focused issue exists, fetches compact issue details using the existing read-only issue API path.
- Does not require user to repeat the issue key.

Expected use:

- User says “inspect focused issue”, “plan current issue”, “這張票怎麼改”, or similar.

## Session persistence

Use `pi.appendEntry()` with a custom type, for example:

```text
jira-board-context
```

The entry is for extension state only and does not participate in LLM context.

On session start:

- Walk `ctx.sessionManager.getBranch()`.
- For matching custom entries, apply them in order.
- Latest branch-local state wins.

## Board handling in M3.1

Minimal board context only:

- If config has `boardId`, current context reports it.
- Existing board snapshot behavior remains available.
- Issue browser may continue project/facet-based narrowing for now.

Board picker and richer board filter/sprint selection are deferred to optional M3.3.
