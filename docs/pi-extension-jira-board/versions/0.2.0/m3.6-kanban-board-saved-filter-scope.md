## Objective

Use Kanban board saved filters as issue browsing scope. Selecting a Kanban board should let `/jira-issues` begin from the board's saved filter JQL, then apply user-selected facets.

## Context

Scrum boards can narrow issue search through active sprint scope. Kanban boards usually narrow issue search through their saved filter. M3.6 validates the Jira Server/Data Center API shape and implements safe-enough JQL composition for this use case.

## Scope

M3.6 includes:

1. Fetch board configuration for active Kanban board.
2. Resolve saved filter id/name from board configuration.
3. Fetch saved filter JQL.
4. Store board filter scope in active board context.
5. Show board filter scope in cockpit and `jira_get_current_context`.
6. Make `/jira-issues` default scope use board saved filter JQL.
7. Compose faceted filters with board saved filter JQL.
8. Preserve/handle existing `ORDER BY` clauses.
9. Fall back gracefully to project scope if configuration/filter lookup fails.

## Non-goals

- Full JQL parser.
- Custom field metadata system.
- Board column/swimlane rendering.
- Jira issue movement.
- Jira writes or LLM-facing write tools.

## API candidates

```text
GET /rest/agile/1.0/board/{boardId}/configuration
GET /rest/api/2/filter/{filterId}
```

Expected board configuration may include:

```json
{
  "filter": {
    "id": "10001",
    "name": "Web Kanban Filter"
  }
}
```

M3.6 should verify the real Jira Server/Data Center shape and add compatibility tests around the observed shape.

## JQL composition

Need helper functions such as:

```ts
splitJqlOrderBy(jql): {
  where: string;
  orderBy?: string;
}

composeScopedJql({
  baseJql,
  clauses,
  fallbackOrderBy,
})
```

Example:

```text
base: project = CHATAPP ORDER BY Rank ASC
clauses: component = IOS, statusCategory != Done
```

Result:

```jql
(project = CHATAPP) AND component = IOS AND statusCategory != Done ORDER BY Rank ASC
```

If no original order exists:

```jql
(project = CHATAPP) AND statusCategory != Done ORDER BY updated DESC
```

This is intentionally not a full parser, but should be tested for common Jira board filter shapes.
