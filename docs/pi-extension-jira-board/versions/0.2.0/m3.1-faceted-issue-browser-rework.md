## Objective

Replace the sequential `/jira-issues` guided prompts with a faceted issue browser: users choose a filter type, then choose actual available values. This keeps the active project context from M3.1 but makes issue filtering usable without remembering Jira fields, field values, or JQL.

## Background

The first M3.1 implementation improved defaults and added guided prompts, but UX review found it still too high-friction:

- Users still need to know fields such as fixVersion/component/assignee.
- Users still need to type values from memory.
- The browser should expose relevant filters and values contextually.

The revised direction is:

```text
Issue browser → Add filter → Choose filter type → Choose actual value → Refresh issue list
```

Raw JQL remains available only as an advanced escape hatch.

## SPEC

### Scope

M3.1 revised includes:

1. Keep existing active project context behavior.
2. Add lightweight project metadata helpers:
   - `GET /rest/api/2/project/{projectKey}/versions`
   - `GET /rest/api/2/project/{projectKey}/components`
3. Add issue result aggregation for filter values:
   - labels
   - assignees
   - statuses
   - priorities
   - issue types
4. Change `/jira-issues` filter action from sequential prompts to faceted selection.
5. Maintain a guided filter state while browsing issues.
6. Rebuild JQL from the current filter state.
7. Keep `/jira-issues --jql <JQL>` for advanced raw JQL mode.
8. Tests for metadata paths, aggregation, filter JQL, and browser state helpers.

### Non-goals

- Full Jira metadata system.
- Metadata cache beyond simple per-command/runtime reuse.
- Custom field support.
- Persisting filters.
- Saving active project to config.
- Clickable widgets.
- Jira writes or LLM-facing write tools.

### Data sources

Use these first:

1. **Project versions API** for fixVersion values:

```text
GET /rest/api/2/project/{projectKey}/versions
```

2. **Project components API** for component values:

```text
GET /rest/api/2/project/{projectKey}/components
```

3. **Issue result aggregation** from the current issue page for:

- labels
- assignees
- statuses
- priorities
- issue types

If a value source is empty, show a clear disabled/empty option and keep the user in the browser.

### Facet types

Initial facet types:

- Fix Version → project versions API → `fixVersion = <value>`
- Component → project components API → `component = <value>`
- Label → issue aggregation → `labels = <value>`
- Assignee → issue aggregation → `assignee = <value>`
- Status → issue aggregation → `status = <value>`
- Priority → issue aggregation → `priority = <value>`
- Issue Type → issue aggregation → `issuetype = <value>`
- Text Search → prompt text → `text ~ <value>`
- Status Category → select not done/all/done → `statusCategory ...`
- Advanced JQL → prompt raw JQL, switches to advanced mode

### `/jira-issues` browser behavior

Default:

- Resolve project from active project, command shorthand, or configured project.
- Start with filter state:

```ts
{
  project: "CHATAPP",
  statusMode: "notDone",
  order: "updatedDesc"
}
```

Keyboard/action behavior:

- `↑/↓` select issue.
- `Enter` focus issue into widget.
- `n`/`p` page.
- `f` or `/` add filter.
- `c` clear non-project filters.
- `Esc` cancel.

Keep `/` accepted for backwards compatibility with the existing picker, but display help should mention `f filter`.

### Widget behavior

Widget remains a read-only context cockpit:

- active project
- filter summary
- issue count/page context
- focused issue card
- suggested actions

No click interaction is expected.
