
- `/jira-fix [KEY]`
- `/jira-comment [KEY]`

- Jira write commands still require interactive UI, preview, and confirmation.
- No LLM-facing write tools are added.

## Objective

Consolidate the M3 Jira UX around a smaller command model and one canonical Jira context cockpit widget. The goal is to reduce scenario-specific widget output and make project/board/filter/focused issue state feel coherent.

## Problem

M3/M3.1 added useful pieces, but the UX is now spread across several commands and widget formats:

- `/jira-projects` displays a selected project card.
- `/jira-issues` displays focused issue cards.
- `/jira-issue` displays compact issue context.
- `/jira-refresh` displays board snapshot context.
- `/jira-status` displays configuration/connectivity context.

This works technically, but it creates too many widget scenarios. Users should be able to understand one persistent Jira cockpit instead.

## Design principles

1. Keep a single Jira widget/cockpit mental model.
2. Widget is read-only context display, not a clickable Jira UI.
3. Primary interaction remains slash commands and `ctx.ui.custom()` browser overlays.
4. Context hierarchy is:

```text
project > board > issue filters > focused issue
```

5. Widget rendering should only have two main modes:
   - compact
   - focus
6. Avoid removing commands abruptly; first consolidate behavior and documentation.

## Command taxonomy

### Setup / health

```text
/jira-onboarding
/jira-status
```

- `/jira-onboarding` configures Jira.
- `/jira-status` validates config/connectivity and updates cockpit status.

### Browse / context

```text
/jira
/jira-projects [query]
/jira-issues [project | --jql <JQL>]
/jira-issue [KEY]
/jira-refresh
```

Recommended behavior:

- `/jira` becomes the primary hub/cockpit command, if scope remains small.
- `/jira-projects` selects active project and updates cockpit.
- `/jira-issues` browses/facets issues and updates filter/focused issue context.
- `/jira-issue <KEY>` inspects an issue and also sets it as focused issue.
- `/jira-issue` without key may inspect focused issue if one exists.
- `/jira-refresh` refreshes cockpit snapshot/status instead of producing a separate widget shape.

### Planning and write actions

```text
/jira-plan [KEY]
/jira-fix [KEY]
/jira-comment [KEY]
/jira-transition [KEY]
```

Future direction, not required for M3.4 unless trivial:

- If key is omitted, use focused issue.
- Keep write actions interactive and confirmation-gated.

## Canonical cockpit state

Conceptual state:

```ts
interface JiraCockpitState {
  project?: {
    key: string;
    name?: string;
  };
  board?: {
    id: number;
    name?: string;
    scope?: "active sprint" | "saved filter" | "none";
  };
  filter?: {
    summary: string;
    total?: number;
    returned?: number;
    startAt?: number;
  };
  focusedIssue?: {
    key: string;
    summary: string;
    status: string;
    priority?: string;
    assignee?: string;
  };
  status?: {
    kind: "ok" | "warning" | "error";
    message: string;
  };
}
```

This state should be derived from or aligned with the existing Jira runtime context from M3.1.

## Widget modes

### Compact mode

Default mode for project/filter/status updates.

Example:

```text
Jira · CHATAPP · Board none
Filter: not done · component:IOS
Issue: CHATAPP-5410 In Progress
Actions: /jira · /jira-issues · /jira-refresh
```

If unavailable/error:

```text
Jira · unavailable
Status: Missing Jira configuration: ...
Actions: /jira-onboarding
```

### Focus mode

Used when an issue is explicitly focused or inspected.

Example:

```text
Jira · CHATAPP · Board none
Filter: not done · component:IOS

CHATAPP-5410  In Progress  Medium
[iOS] 既有功能 API 重構為 core module 模式_Phase 2
assignee: anton_liu

Actions: /jira-plan · /jira-fix · /jira-transition
```

## Command behavior matrix

| Command | State updates | Widget mode | Notes |
| --- | --- | --- | --- |
| `/jira` | none or status refresh | compact | Optional hub/cockpit entrypoint |
| `/jira-projects` | active project | compact | Persist context entry |
| `/jira-issues` filter change | filter summary/counts | compact | Faceted browser remains primary interaction |
| `/jira-issues` issue focus | focused issue | focus | Persist context entry |
| `/jira-issue <KEY>` | focused issue | focus | Direct inspect should match browser focus |
| `/jira-refresh` | snapshot/status/counts | compact | No separate snapshot widget shape |
| `/jira-status` | status | compact | No separate status widget shape |

## Scope

M3.4 includes:

1. Add canonical cockpit state/renderer helpers.
2. Route widget updates through the cockpit renderer.
3. Support compact/focus rendering modes only.
4. Ensure `/jira-projects`, `/jira-issues`, `/jira-issue`, `/jira-refresh`, and `/jira-status` use the same widget model.
5. Make `/jira-issue <KEY>` set focused issue state.
6. Optionally add `/jira` hub command if it stays simple.
7. Update docs and tests for command taxonomy and widget formats.

## Non-goals

- Removing existing commands.
- Board picker.
- Full Jira dashboard UI.
- Clickable widget interactions.
- New Jira write behavior.
- Persisting active project/board to local config.

## AC

M3.4 is complete when:

1. There is one canonical Jira cockpit widget renderer.
2. Widget output has only compact/focus modes for normal states.
3. Existing commands no longer each render unrelated widget layouts.
4. `/jira-issue <KEY>` sets focused issue context.
5. `/jira-refresh` and `/jira-status` update cockpit status/snapshot without replacing the widget mental model.
6. `jira_get_current_context` remains consistent with cockpit state.
7. No automatic context injection is added.
8. Tests and typecheck pass.
