# M3.2 optional plan — Jira metadata system

## Status

Optional / deferred.

## Objective

Evaluate and potentially build a fuller Jira metadata system after the faceted browser UX proves useful. This is intentionally not required for v0.2.0 unless the user decides it is worth the scope.

## Context

M3/M3.1 discussion clarified that `/jira-issues` should not require users to know JQL or remember filter fields/values. The preferred UX is a faceted browser:

```text
Add filter → choose filter type → choose actual value
```

For the first faceted implementation, use a pragmatic data model:

- Project versions from Jira project versions API.
- Project components from Jira project components API.
- Labels/status/priority/issue type/assignee from issue result aggregation.

A full metadata system is deferred because Jira metadata APIs vary and can expand scope quickly.

## Optional scope

If M3.2 is activated, it may include:

1. Central metadata cache keyed by base URL + project.
2. Project metadata fetching helpers.
3. Field value providers for:
   - versions
   - components
   - labels
   - statuses
   - priorities
   - issue types
   - assignees
4. Cache invalidation and refresh command.
5. Metadata-aware filter picker.
6. Tests for metadata mapping, caching, fallback behavior, and stale/error states.

## Non-goals unless explicitly promoted

- Supporting every custom Jira field.
- Schema-driven Jira form rendering.
- Jira issue creation/edit forms.
- Cross-project global metadata indexing.
- Long-lived persistent metadata cache.

## Decision gate

Before implementing M3.2, evaluate:

- Is the lightweight faceted browser insufficient?
- Are users frequently missing values not visible from project versions/components plus issue aggregation?
- Does metadata fetching add noticeable latency?
- Is the UX worth the complexity before v0.2.0 release?

Default decision: **defer**.
