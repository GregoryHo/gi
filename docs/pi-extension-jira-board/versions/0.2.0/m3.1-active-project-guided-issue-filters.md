# M3.1 implementation plan — Active project context and guided issue filters

## Status

## Post-implementation UX revision

After implementation, the guided sequential prompt approach was judged still too high-friction. The desired UX is now:

- `/jira-projects` applying active project remains correct.
- `/jira-issues` should use a faceted filter browser rather than sequential manual field prompts.
- Users should not need to know filter fields like fixVersion/component/label ahead of time.
- The browser should show relevant filter types and actual available values.
- First data source should be project versions/components API plus aggregation from issue results.
- Full metadata system is optional and deferred to M3.2.

## Objective

Polish the M3 browse UX so selecting a project actually changes the active Jira context, and users can browse issues with guided filters instead of needing to write raw JQL.

## Problem

M3 added `/jira-projects` and `/jira-issues`, but two UX gaps remain:

1. Selecting a project is only session-local for `/jira-issues`; it does not affect autocomplete, default issue search, board snapshot, or refresh behavior.
2. `/jira-issues` currently treats command args and filter changes as raw JQL, which creates too much cognitive load for normal users.

## SPEC

### Scope

M3.1 includes:

1. Runtime active Jira context shared across extension features.
2. `/jira-projects` selection applies the selected project as active project for the current session.
3. Existing default project behavior uses active project first, then config/env project.
4. Autocomplete uses active project for preload and `#number` shorthand where possible.
5. `/jira-issues` uses guided filters by default.
6. Raw JQL remains available as an advanced escape hatch.
7. Tests for active context resolution, guided JQL builder, and affected default behavior.

### Non-goals

M3.1 does not include:

- Persisting selected project to local config by default.
- Asking whether to save selected project permanently.
- Project/board mapping or board auto-selection.
- Rich component/fixVersion autocomplete.
- Jira writes or LLM-facing write tools.

### Active project context

Introduce an extension-local runtime context, likely a small module:

```text
packages/pi-extension-jira-board/src/jira-context.ts
```

Conceptual state:

```ts
interface JiraRuntimeContext {
  activeProject?: JiraProject;
}
```

Behavior:

- `/jira-projects` selection sets `activeProject`.
- Active project is session/runtime-local only.
- Active project is not written to `config.json` in M3.1.
- Future work may add explicit “save as default project” confirmation.

Resolution order for default project:

```text
active project > configured project > undefined
```

Affected features:

- `/jira-issues` default issue search.
- `jira_search_issues` default JQL when no `jql` provided.
- `jira_board_snapshot` and `/jira-refresh` project fallback.
- Jira autocomplete project-key preload and `#number` shorthand.

Implementation note:

Because tools currently call `loadJiraConfig()` directly, prefer a small helper that applies runtime context to a loaded config:

```ts
applyJiraRuntimeContext(config): JiraConfig
```

Keep this minimal and avoid broad refactors.

### Guided issue filters

`/jira-issues` should no longer require users to know JQL for normal browsing.

Default flow:

- If command args are empty, use guided filter state.
- Initial project defaults to active/default project.
- Default status behavior: not done.
- Default order: updated desc.

Guided filter fields:

- project key — required unless config/active project exists.
- fixVersion — optional.
- component — optional.
- assignee — optional.
- text search — optional.
- status mode:
  - not done (default): `statusCategory != Done`
  - all: no status clause
  - done: `statusCategory = Done`
- order:
  - updated desc (default)

JQL builder examples:

```text
project = CHATAPP AND statusCategory != Done ORDER BY updated DESC
project = CHATAPP AND fixVersion = "v1.62" AND component = IOS AND statusCategory != Done ORDER BY updated DESC
project = CHATAPP AND text ~ "login" AND statusCategory != Done ORDER BY updated DESC
```

Quoting rules:

- Quote values that contain spaces, dots, or special characters.
- Keep simple Jira identifiers unquoted when safe, e.g. `component = IOS`.
- Escape embedded quotes in quoted values.

### `/jira-issues` command UX

Recommended behavior:

- `/jira-issues` opens issue browser using active/default project and default guided filters.
- `/jira-issues CHATAPP` may be treated as project shorthand, not raw JQL.
- `/jira-issues --jql project = CHATAPP ...` uses raw JQL advanced mode.
- Inside the picker, `/` opens guided filter prompts instead of raw JQL input.
- Advanced raw JQL can be available through a separate prompt option if needed, but not the default path.

Keep M3.1 implementation simple: use sequential `ctx.ui.input()` / `ctx.ui.select()` prompts for guided filters rather than building a complex custom form.

### Autocomplete behavior

Autocomplete should prefer active project for:

- recent issue preload JQL.
- `#123` shorthand expansion.

If no active project exists, keep existing configured project behavior.
